# HTML Triage Report Design

**Date:** 2026-03-29
**Status:** Approved
**Author:** Ognjen Popovic

## Overview

Add a self-contained HTML report as an additional output of `/taskflow:triage`. The report provides a visual, layered view of triage results that's easier to review than the terminal summary or markdown file.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Dependency on other plugins | None — self-contained | Users shouldn't need `frontend-design` or any other plugin |
| Interactivity | Read-only (v1) | Interactive state writes add complexity; read-only solves 80% of the pain |
| Information density | Layered progressive disclosure | Stats at top, batches expandable, tasks within batches expandable |
| File delivery | Write file, print path | Auto-open is annoying in headless/repeated-run scenarios |

## Output

The triage skill writes one additional file:

```
<config.output_dir>/triage-report.html
```

This file is overwritten on each triage run (not date-stamped — there's only ever one current report). It's a self-contained HTML file with all CSS and JS inline. No external dependencies. Opens in any browser.

## Data Source

The report is generated from the same data the skill already has in memory after classification and batching. It uses the exact same data that goes into the state JSON and markdown summary:

- Developer name, triage date
- Per-task: id, title, url, area, type, complexity, confidence, implementable, time estimates (human + agent), files involved, approach steps
- Per-batch: name, task list, suggested branch, total time estimates
- Aggregate stats: total tasks, batch count, area distribution, total times

No extra data fetching or processing is needed.

## Layout Structure

### 1. Header
- Report title: "Triage Report"
- Meta line: developer name, date, total task count, batch count

### 2. Stats Bar
Four stat cards in a horizontal row:
- **Tasks Triaged** (green) — total count
- **Batches** (blue) — batch count
- **Agent Time** (yellow) — total estimated agent time across all tasks
- **Human Time** (purple) — total estimated human time across all tasks

### 3. Area Distribution Bar
A horizontal segmented bar showing the proportion of tasks per area. Each segment is a different color. Legend below with area names and counts.

Colors are assigned dynamically based on the areas in the config — not hardcoded to specific area names. Use a predefined palette of 8-10 colors and assign them to areas in order.

### 4. Batch List
Each batch is a collapsible card:

**Batch header (always visible):**
- Chevron indicator (expand/collapse)
- Batch number badge
- Batch name (e.g., "Order email fixes")
- Task count, agent time total, human time total
- Suggested branch name (monospace)

**Batch body (expanded):**
- List of task rows, each collapsible
- Batch totals footer row with time sums and branch name

### 5. Task Cards (within batches)
Each task is a collapsible row within a batch:

**Task header (always visible):**
- Chevron indicator
- Task title
- Badges: type (bug/feature/etc), complexity (small/medium/large), confidence (high/medium/low), area

Badge colors:
- Type: bug=red, feature=blue, copy-change=gray, investigation=yellow
- Complexity: small=green, medium=yellow, large=red
- Confidence: high=green, medium=yellow, low=red
- Area: purple (all areas use the same badge color for consistency)

**Task detail (expanded):**
- Time estimates: human and agent
- Task link: clickable link to the task in the provider's UI (opens new tab)
- Files involved: list of file paths in code formatting
- Approach: numbered steps from the classification
- Risks/unknowns: if any were identified

### 6. Footer
- "Generated by taskflow" with the output file path

## Interaction

- Click a batch header to expand/collapse it
- Click a task header to expand/collapse it (independent of batch state)
- First batch starts expanded, first task within it starts expanded — so the report isn't a wall of collapsed cards on load
- All other batches and tasks start collapsed
- No other interactivity — no forms, no state writes, no drag-and-drop

## Visual Design

- Dark theme (GitHub dark style) — matches the mockup approved during brainstorming
- Color palette: bg `#0d1117`, surface `#161b22`, border `#30363d`, text `#e6edf3`, muted `#8b949e`
- Accent colors: blue `#58a6ff`, green `#3fb950`, yellow `#d29922`, red `#f85149`, purple `#bc8cff`, orange `#db6d28`
- Max width: 1100px, centered
- Smooth chevron rotation on expand/collapse (CSS transition)
- Hover state on clickable headers (subtle background change)
- Monospace for branch names and file paths

## Integration into Triage Skill

The HTML report is generated as a new **Step 8.5** in the triage workflow, between "Write Summary File" (Step 8) and "Print Terminal Summary" (Step 9).

### Step 8.5 — Generate HTML Report

After writing the summary file, generate the HTML report:

1. Build the HTML string from the triage data already in memory (classifications, batches, state)
2. The HTML is self-contained: all CSS in a `<style>` block, all JS in a `<script>` block, no external resources
3. Write to `<config.output_dir>/triage-report.html`
4. In Step 9 (terminal summary), add the report path to the file locations output:
   ```
   Report file:  <config.output_dir>/triage-report.html
   ```

### Skill Instructions for HTML Generation

The triage SKILL.md needs to include:
- The complete HTML template structure (header, stats, area bar, batch list, task cards, footer)
- CSS variables and styles
- The JS for expand/collapse toggling
- Instructions for how to populate each section from the triage data
- The color palette and badge styling rules
- Dynamic color assignment logic for the area distribution bar

The mockup file at `.superpowers/brainstorm/525260-1774782670/content/triage-report-mockup.html` serves as the reference implementation. The skill instructions should produce output visually matching this mockup.

## What Does NOT Change

- The existing 9-step workflow is unaffected (Step 8.5 is additive)
- The markdown summary still gets written
- The state JSON still gets written
- The terminal summary still gets printed
- No new config fields needed — the output directory is already configured
